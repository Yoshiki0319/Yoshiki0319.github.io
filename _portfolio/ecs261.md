This project explores the **formal modeling and verification** of the TCP **3-way handshake**, the **SYN Flooding attack**, and a **SYN cookie-based defense** using **Dafny** and **Python**. The work focuses on proving the correctness of the handshake state transitions, demonstrating how backlog exhaustion occurs under adversarial SYN traffic, and showing how SYN cookies can remove the need for server-side backlog state. ([Github](https://github.com/Yoshiki0319/ECS261-Project))

### Main components

1. **3-way handshake formal model**
   The project models the standard TCP connection-establishment process between client and server, including packet representation, connection states, and server-side transition rules. In the verified model, the server accepts a SYN, inserts a half-open connection into the backlog, responds with SYN-ACK, and establishes the connection when a matching ACK arrives.  

2. **SYN Flooding attack model**
   The project extends the normal handshake model with adversarial behavior in which attackers repeatedly send SYN packets without returning the final ACK. This causes half-open connections to accumulate in the finite backlog until capacity is exhausted, preventing legitimate clients from connecting. The attack is modeled formally in Dafny and also reproduced in Python through concurrent attacker simulation.  

3. **Backlog and server-state verification**
   A key contribution of the project is verifying the correctness of backlog manipulation. The authors prove that valid SYN packets create half-open entries when space is available, that valid ACKs remove the corresponding entries, and that repeated incomplete handshakes monotonically consume finite server resources until saturation. 

4. **SYN cookie model**
   As a reach goal, the project implements a backlog-free handshake model augmented with SYN cookies. Instead of storing intermediate state on the server, the server generates a cookie and validates it when the client replies. The verified result shows that a connection is established only when the ACK matches the expected cookie, while incomplete handshakes leave no server-side state behind.  

5. **Python + Dafny integration**
   The workflow combines Python and Dafny: the team first prototyped behavior in Python, then implemented and verified the abstract models in Dafny, and finally compiled Dafny code into Python modules using `dafny build --target:py`. Python was also used to simulate multi-threaded attackers because Dafny itself does not support threads.  

### Results

The project achieved three layers of outcomes. First, it verified the correctness of the simplified TCP 3-way handshake under explicit preconditions and postconditions. Second, it verified that repeated incomplete SYN requests can saturate the server backlog and create a denial-of-service condition. Third, it showed that a SYN cookie design can eliminate backlog state while still allowing only valid clients to complete the handshake. Figures in the report illustrate the handshake flow, attack scenario, verification traces, and backlog-free SYN cookie execution.  

### Challenges

A major challenge was the mismatch between the desired attack model and Dafny’s capabilities. Since Dafny is fundamentally single-threaded, the concurrent flooding aspect had to be simulated in Python instead of verified directly in Dafny. Another challenge was modeling a true cryptographic hash for SYN cookies; the report explains that modular arithmetic, large primes, and collision reasoning were too difficult to verify within the project scope, so the cookie mechanism was simplified.  

### Takeaway

This project shows how **formal methods** can be applied to a practical cybersecurity problem. Rather than building a full production TCP stack, the work isolates the essential resource-allocation logic behind TCP connection setup and SYN flooding, then verifies both the vulnerability and its defense at the abstract model level. The result is a strong example of combining **systems security**, **protocol modeling**, and **formal verification** in a compact research-oriented implementation.
